# 13 - Inverse kinematics (ENTRY NOT YET COMPLETE)

I wanted to get inverse kinematics to work on the arm as well. I wanted to be able to specify an (x, y, z) position for the tip of the gripper to be at and calculate the joint angles to move there. This is the reverse of forward kinematics, where you already know every joint angle and just chain the link transforms down the arm to find where the tip lands. Forward kinematics has one clean answer; inverse kinematics is the harder direction, since a given target position has no tidy closed-form solution on an arm shaped like this one and generally has to be solved numerically.

The arm is technically 6-DOF, but the wrist roll and gripper open do not technically change the end tip position — the wrist roll just spins the gripper about its own axis and the gripper open only moves the jaws — so I hold both of them fixed and leave them on their manual sliders. That leaves us trying to solve for 4 variables (shoulder pan, shoulder lift, elbow flex, wrist flex) given 3 inputs (x, y, z). Having more joints to solve than numbers to satisfy means the arm is redundant: there is not a single answer but a whole family of joint configurations that all place the tip at the same point. The solver has to pick one of them, which it does by preferring the solution that stays closest to where the arm already is, so the motion comes out smooth instead of jumping to some far-off configuration that happens to reach the same spot.

I was able to reuse the [control layer I previously created for slider control](https://github.com/sudhir-c/so101-journal/blob/main/journal/10_teleopdashboard.md) so all that had to be done was define an interface to specify target angles and solve for the joint positions. The inverse kinematics only ever produces a dictionary of target joint angles, and those get handed straight to the same safety core I built for the sliders — the per-joint clamping to the calibrated limits, the velocity limit that caps how fast any joint can move, and the STOP button all still apply, with no changes to that layer at all. That also gave me a clean way to handle targets that are out of reach. When a point is unreachable the solver still returns its best attempt, which sits at the edge of the workspace rather than on the target, so before moving anything I solve, clamp the result to the real joint limits, run forward kinematics on that clamped solution, and check whether the resulting tip actually lands within a few millimetres of what I asked for. If it does not, the arm just holds still instead of creeping toward the boundary of what it can do.

I had the zero position default to the current position of the arm upon control server startup, and made it resettable so it was easy to control between sessions. When the server starts it reads the current joint angles, runs forward kinematics to work out where the tip currently is, and stores that point as the (0, 0, 0) origin. Every target I then type in is an offset in millimetres from that origin, and a reset button re-reads the current tip and makes it the new zero whenever I want to recenter, which is handy since the arm does not always come up in exactly the same pose.

LeRobot defines its own solver, based on the [Placo](https://placo.readthedocs.io/en/latest/#) solver. We cannot use LeRobot's solver directly for a few reasons, but we can copy over most of the logic. The first reason is the middle joint. LeRobot's wrapper solves for whichever joints you give it and can only hold trailing joints fixed, so it has no way to freeze the wrist roll, which sits in the middle of the chain rather than at the end. If the wrist roll is left free, the solver will use it to help reach the target and hand back a four-joint answer that is only valid if the wrist roll had also moved, which defeats the whole point of keeping it on a manual slider. So rather than use the wrapper I drive Placo directly and add the wrist roll and gripper as a hard constraint pinned to their current values, which leaves the solver free to move only the four joints I actually want it to. The second reason is an environment conflict. Placo depends on a newer version of numpy than the one my training and LeRobot stack is pinned to, and installing it into the main environment would drag numpy up and risk breaking the policy training setup, so I run the solver in its own isolated environment as a small subprocess that the control server talks to over a pipe. The third reason is that the solver needs an actual geometric model of the robot, a URDF, and LeRobot does not ship one for the SO-101, so I pulled the official SO-ARM100 URDF and stripped it down to just the kinematic chain — the joints, links, and their offsets — since the solver has no need for the visual meshes.

The logic I did copy over is the core Placo setup: load the URDF, fix the base of the robot in place, put a position-only task on the gripper tip frame so the solver only cares where the tip ends up and not which way the wrist is turned, seed it with the arm's current joint angles, and then iterate the solver until the tip converges onto the target. Everything is handled in degrees on the outside to match the rest of the control stack and converted to radians for Placo internally. Because the solve runs on every update at around thirty times a second and each one only takes about a millisecond, it comfortably keeps up with live dragging of the target, and the velocity limit in the control layer smooths out any larger jumps in the solution.
